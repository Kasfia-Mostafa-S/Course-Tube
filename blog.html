Question: Discuss the scope of var, let, and const.

Ans : In JavaScript, there are some differences between var, let, and const variable declarations. The scope of variables declared with the var and const keywords is the current function body. The var keyword declares variables that are hoisted. Hoisting describes the ability to access a variable in its surrounding scope even before it has been declared. Let-declared variables are block-scoped, which means that they only apply to the current block in which they are declared.
Function scope applies to variables declared with the var keyword. When something is function scoped, it can only be accessible within the function in which it was declared.Hoisting applies to variables that are declared using the var keyword. The JavaScript runtime will move a variable to the top of its scope if we declare it (but do not initialize it) at the conclusion of the function. As a result, the runtime will not object if we use the variable before it has been declared.
When you want to declare a variable that should only be accessible within the block in which it is declared, you should use the let keyword. Furthermore, variables created using the let keyword cannot be changed or re-created.The let keyword is used to declare variables that are not subject to hoisting. This signifies that a variable cannot be used until it has been declared and initialized.
Similar guidelines apply to the let keyword and const. Const is similar to other types of definitions, however in JavaScript programs, const is only used to define constant values.The let keyword and const both use the same scoping concepts. Similar to let, the scope of variables declared using the const keyword will be limited to the block in which they are declared.

Question: What do you mean by REST API?.

Ans : Representational State Transfer Application Programming Interface is referred to as REST API. It is a method of designing networked applications. REST is a collection of guidelines and restrictions that determine how online services should be organized and communicate with one another, not a particular technology or protocol.
REST isolates the client (the API consumer) from the server (the API provider), allowing each to develop on its own. This division of duties encourages system scalability and makes development easier.
Whenever a client sends a request to a server, the server must get all the information required to comprehend and handle the request. In-between requests, the client's status shouldn't be recorded by the server. Due to their high scalability and simplicity, REST APIs.
Resources in REST are denoted by nouns (e.g., /users, /products), and they can be altered using conventional HTTP techniques. The data objects or entities that the API exposes can be considered resources.
The information required to process each request made by a client to a server must be self-contained in the request and included in it. It is not advisable for the server to interpret current requests based on prior requests or sessions.
RESTful interactions are stateless, therefore each request sent by a client to a server must include all the data required to comprehend and fulfill it. The server shouldn't process the current request based on previous ones.
